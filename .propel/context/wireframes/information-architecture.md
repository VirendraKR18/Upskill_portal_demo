## 1. Wireframe Specification

**Fidelity Level**: High (Production-Ready Mockups)
**Screen Type**: Responsive Web (Desktop, Tablet, Mobile)
**Viewport**: Desktop 1440px, Tablet 768px, Mobile 375px
**Framework**: React 18 + TypeScript + Tailwind CSS
**Design System**: See [designsystem.md](../docs/designsystem.md)

## 2. System Overview

The AI-Powered Credit-Based Learning Platform is an enterprise learning management system designed to systematically upskill organizational resources in AI technologies. The platform features a credit-based progression system, gamification elements (badges, leaderboards), certification workflows, and role-based dashboards for Learners, Managers, Admins, and Leadership.

**Key Functionalities:**
- SSO authentication with role-based routing
- Learning path enrollment and course completion tracking
- Credit earning and verification system
- Global and team leaderboards with real-time rankings
- Certification application and approval workflows
- Team progress monitoring and skill gap analysis
- Content management and credit audit tools
- Executive AI readiness dashboards and reporting

## 4. User Personas & Flows

### Persona 1: Learner (Engineering Team Member)
- **Role**: Individual contributor seeking AI upskilling
- **Goals**: Earn credits, complete certifications, track progress, compete on leaderboards
- **Key Screens**: SCR-001 (Login), SCR-002 (Dashboard), SCR-003 (Leaderboard), SCR-004 (Profile), SCR-005 (Learning Path Browser), SCR-006 (Certification Application), SCR-010 (Course Player), SCR-018 (Path Detail), SCR-019 (Course Detail), SCR-020 (Assessment), SCR-021 (Project Submission), SCR-022 (Team Leaderboard), SCR-023 (Settings), SCR-024 (Cert Status)
- **Primary Flow**: Login → Dashboard → Browse Paths → Enroll → Course Player → Assessment → Credits Awarded → Leaderboard
- **Decision Points**: 
  - Prerequisites met for path enrollment?
  - Assessment score ≥70% for credit?
  - Apply for external certification?

### Persona 2: Manager (Team Lead/Dept Manager)
- **Role**: People manager monitoring team skill development
- **Goals**: Track team progress, approve certifications, identify skill gaps, recommend learning paths
- **Key Screens**: SCR-001 (Login), SCR-007 (Manager Dashboard), SCR-011 (Team Member Detail), SCR-012 (Certification Approval Queue), SCR-022 (Team Leaderboard), SCR-025 (Cert Review), SCR-026 (Skill Gap Analysis)
- **Primary Flow**: Login → Manager Dashboard → Review At-Risk Learners → View Team Member Detail → Approve Certification → Skill Gap Analysis
- **Decision Points**:
  - Is team member at-risk (no activity in 14 days)?
  - Approve or reject certification application?
  - Assign learning path to close skill gap?

### Persona 3: Admin (Platform Administrator)
- **Role**: Content manager and platform administrator
- **Goals**: Manage learning content, configure credit rules, audit credit integrity, ensure compliance
- **Key Screens**: SCR-001 (Login), SCR-008 (Admin Console), SCR-013 (Content Management), SCR-014 (Credit Audit), SCR-015 (Anomaly Detection), SCR-027 (Create Path), SCR-028 (Create Course), SCR-029 (Upload Materials), SCR-030 (Transaction Detail), SCR-031 (Anomaly Investigation)
- **Primary Flow**: Login → Admin Console → Create Learning Path → Create Course → Upload Materials → Publish → Monitor Credit Audit
- **Decision Points**:
  - Approve or flag credit transaction?
  - Investigate detected anomaly?
  - Publish or keep course in draft?

### Persona 4: Leadership (C-Level Executives)
- **Role**: Strategic decision-maker for AI transformation
- **Goals**: View AI readiness metrics, allocate training budgets, track ROI, make strategic training decisions
- **Key Screens**: SCR-001 (Login), SCR-009 (Leadership Dashboard), SCR-016 (Executive Reports), SCR-032 (Report Export)
- **Primary Flow**: Login → Leadership Dashboard → Review AI Readiness Score → Analytics by Department → Export Report
- **Decision Points**:
  - Which departments need priority AI investment?
  - Is training ROI positive?
  - Budget allocation for next quarter?

## 7. Interaction Patterns

### Pattern 1: Learning Path Enrollment (FL-002)
- **Trigger**: User clicks "Enroll Now" button on SCR-018 (Learning Path Detail)
- **Flow**: 
  1. System checks prerequisites (backend validation)
  2. If prerequisites met: Show enrollment confirmation modal with path summary
  3. User confirms enrollment (modal CTA)
  4. System enrolls user, updates database
  5. Redirect to SCR-002 (Dashboard) with success toast: "Enrolled in '<Path Name>' successfully!"
  6. Dashboard updates: New path appears in "My Learning Paths", first course in "Recommended Next Steps"
- **Screens Involved**: SCR-018 → SCR-002
- **Feedback**: 
  - Loading spinner on "Enroll Now" button (200ms perceived performance)
  - Success toast notification (auto-dismiss after 5 seconds)
  - Dashboard widgets update with new enrollment
- **Components Used**: Button (Primary), Modal Dialog, Toast Notification, Loading Spinner, Card (Dashboard Widget)

### Pattern 2: Form Validation (Inline)
- **Trigger**: User finishes typing in form field or clicks Submit
- **Flow**:
  1. On blur (field loses focus): Validate individual field
  2. If invalid: Show red border on input + error message below field (text-sm, text-error-500)
  3. If valid: Show green checkmark icon right of input (optional, accessibility consideration)
  4. On form submit: Validate all fields
  5. If any field invalid: Focus first invalid field, prevent submission, show error summary top of form
  6. If all valid: Show loading state on Submit button, proceed with submission
- **Screens Involved**: SCR-001 (Login), SCR-004 (Profile), SCR-006 (Certification), SCR-027 (Create Path), SCR-028 (Create Course)
- **Feedback**: 
  - Inline error messages (immediate, per UXR-501)
  - Error summary banner for multi-field errors
  - Loading state on Submit button (text changes to "Submitting...", spinner icon)
- **Components Used**: Input Field (with error state), Label, Error Message Text, Submit Button (with loading state)

### Pattern 3: Data Table Row Action
- **Trigger**: User clicks action icon/button on data table row (e.g., "View Details", "Approve", "Investigate")
- **Flow**:
  1. User hovers over row: Row background changes to hover state (bg-neutral-50)
  2. User clicks action button: 
     - **Option A (Modal)**: Modal opens centered with row details
     - **Option B (Drawer)**: Drawer slides in from right (desktop) or bottom (mobile) with details
     - **Option C (Navigation)**: Navigate to dedicated detail screen
  3. User interacts with detail view (read, edit, approve, etc.)
  4. User closes modal/drawer or navigates back: Return to table with updated row data
- **Screens Involved**: SCR-007 (Manager Dashboard table), SCR-012 (Approval Queue), SCR-014 (Credit Audit), SCR-015 (Anomaly Detection)
- **Feedback**:
  - Row hover state (visual feedback)
  - Loading spinner during modal/drawer content load
  - Success/error toast after action (e.g., "Certification approved successfully")
  - Table row updates (refresh data without full page reload)
- **Components Used**: Data Table, Table Row, Action Button (icon button), Modal/Drawer, Toast Notification

### Pattern 4: Video Player Progress Tracking
- **Trigger**: User plays video course content on SCR-010 (Course Player)
- **Flow**:
  1. User clicks Play: Video starts, progress tracking begins
  2. Every 5 seconds: Autosave current playback position to backend (background API call)
  3. User pauses/closes: Save final position
  4. User returns to course: Resume from last saved position (show "Resume from <timestamp>?" prompt)
  5. User completes video (watches to 95%+ or end): Mark section as complete, update TOC checkmark
  6. All sections complete → Enable "Take Assessment" button
- **Screens Involved**: SCR-010 (Course Player)
- **Feedback**:
  - Subtle progress auto-save indicator (small checkmark icon, 2 seconds)
  - Resume confirmation prompt on load (if partial progress exists)
  - TOC checkmark updates in real-time (green checkmark icon)
  - "Take Assessment" button state changes from disabled (gray) to enabled (primary blue)
- **Components Used**: Video Player, Progress Bar, Button (state transition), Toast/Subtle Notification, Checkbox (TOC checkmarks)

## 8. Error Handling

### Error Scenario 1: Network Error (Offline/Timeout)
- **Trigger**: API request fails due to network connectivity loss or timeout
- **Error Screen/State**: Non-blocking error toast (top-right corner, red background, error icon, "Network error. Retrying..." message)
- **User Action**: Automatic retry with exponential backoff (3 attempts: 1s, 2s, 4s). If all retries fail, show persistent error banner with manual "Retry" button
- **Recovery Flow**: 
  1. User clicks "Retry" → Re-execute failed request
  2. Or user goes offline mode (if platform supports): Queue actions for sync when online
  3. Or user navigates away: Discard pending action with confirmation prompt if destructive
- **Affected Screens**: All screens with API interactions
- **Components**: Toast Notification (error variant), Error Banner, Retry Button

### Error Scenario 2: Form Validation Error
- **Trigger**: User submits form with invalid or missing required fields
- **Error Screen/State**: 
  - Inline: Red border on invalid fields, error message below field (e.g., "Email address is required" or "Password must be at least 8 characters")
  - Summary: Error banner top of form: "Please correct 3 errors below before submitting"
- **User Action**: Correct invalid fields based on specific error messages
- **Recovery Flow**:
  1. User corrects field → Error message disappears on blur (field re-validation)
  2. User submits again → If all valid, proceed; if still invalid, repeat error display
- **Affected Screens**: SCR-001 (Login), SCR-004 (Profile), SCR-006 (Certification), SCR-027/028 (Create Content)
- **Components**: Input Field (error state), Error Message Text, Error Summary Banner

### Error Scenario 3: Prerequisite Not Met (Enrollment Block)
- **Trigger**: User attempts to enroll in learning path without completing prerequisites
- **Error Screen/State**: SCR-018 (Learning Path Detail) / Error State
  - "Enroll Now" button disabled (grayed out, cursor: not-allowed)
  - Yellow warning banner: "Prerequisites required: You must complete 'Intro to AI' before enrolling in this path. View Prerequisites →"
- **User Action**: 
  - Click "View Prerequisites" link → Scroll to prerequisites section (smooth scroll)
  - Click prerequisite link → Navigate to prerequisite course detail (SCR-019)
  - Complete prerequisite → Return to this path → "Enroll Now" button enabled
- **Recovery Flow**: Complete prerequisites → Re-check → Enrollment enabled
- **Affected Screens**: SCR-018 (Learning Path Detail)
- **Components**: Button (disabled state), Warning Banner, Link, Prerequisites Panel

### Error Scenario 4: Assessment Failed (Score < 70%)
- **Trigger**: User submits assessment with score below passing threshold (70%)
- **Error Screen/State**: SCR-020 (Assessment) / Error State
  - Error modal: "Assessment Failed" (red icon), "Your score: 55% (passing: 70%)"
  - Retry availability: 
    - If attempts < 3: "You can retry in 24 hours (attempt 2 of 3)"
    - If attempts ≥ 3: "You must retake the course before attempting again"
- **User Action**:
  - If retries available: "OK" button → Return to SCR-010 (Course Player), retry button disabled for 24h
  - If no retries: "Retake Course" button → Return to SCR-010 (from start)
- **Recovery Flow**: Wait 24h → Retry assessment OR Retake course → Retry assessment
- **Affected Screens**: SCR-020 (Assessment)
- **Components**: Modal Dialog (error variant), Score Display, Retry Counter, CTA Buttons

### Error Scenario 5: File Upload Error (Size/Type)
- **Trigger**: User attempts to upload file exceeding size limit (e.g., >500MB) or unsupported type
- **Error Screen/State**: SCR-021 (Project Submission) or SCR-029 (Materials Upload) / Validation State
  - File upload zone shows red border
  - Error message below: "File 'project.zip' exceeds max size (500MB). Please compress or split the file."
  - Or: "File type '.exe' not supported. Accepted formats: .pdf, .zip, .mp4"
- **User Action**: Remove invalid file, upload valid file (correct size/type)
- **Recovery Flow**: User selects valid file → Validation passes → Upload proceeds
- **Affected Screens**: SCR-021 (Project Submission), SCR-029 (Materials Upload)
- **Components**: File Upload Dropzone (error state), Error Message, File List (with remove button)

### Error Scenario 6: Session Expired (401 Unauthorized)
- **Trigger**: User session token expires (e.g., after 8 hours of inactivity) while browsing
- **Error Screen/State**: 
  - Blocking modal (cannot be dismissed): "Your session has expired for security. Please sign in again to continue."
  - "Sign In" button (primary CTA)
- **User Action**: Click "Sign In" → Redirect to SCR-001 (Login) with returnUrl parameter
- **Recovery Flow**: Re-authenticate → Redirect back to original page (via returnUrl)
- **Affected Screens**: All authenticated screens
- **Components**: Modal Dialog (blocking, no close button), Primary Button (Sign In)
