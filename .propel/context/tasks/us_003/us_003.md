## Story ID
   * ID Format: US_003

## Story Title
   * Role-Based Access Control for Four Roles

## Description
  * As a platform administrator, I want role-based access control (RBAC) enforced at the API level for four distinct roles (Learner, Manager, Admin, Leadership) with granular permissions and least-privilege principle, so that users can only access resources and perform actions appropriate to their assigned role.

## Acceptance Criteria
  1. **Given** a user authenticates with a Learner role, **When** they request manager-only endpoints (e.g., GET /api/v1/dashboards/manager, POST /api/v1/certifications/{id}/approve), **Then** the API returns HTTP 403 Forbidden and logs the unauthorized access attempt.

  2. **Given** a user authenticates with a Manager role, **When** they request team data, **Then** the API returns only their direct reports' data (filtered by ManagerID) and blocks access to other teams' data.

  3. **Given** a user authenticates with an Admin role, **When** they access admin endpoints (content management, credit audit, anomaly detection), **Then** all admin operations are permitted and each action is recorded in the AuditLog.

  4. **Given** a user authenticates with a Leadership role, **When** they access executive dashboards, **Then** they see organization-wide aggregated data but cannot perform admin operations (content CRUD, credit adjustments).

  5. **Given** authorization policies are configured, **When** any API request is received, **Then** the middleware extracts the role from the JWT claims, evaluates the policy, and enforces access before the controller executes.

  6. **Given** role assignments change in Workday, **When** the next sync updates a user's role, **Then** the new permissions take effect on the user's next authentication (JWT refresh).

## Edge Cases
   * What happens when a JWT contains multiple roles? (Most permissive role applied; log edge case for review)
   * How does the system handle a request with a valid JWT but deleted/deactivated user? (Middleware checks user active status against database; rejects with 401 if inactive)
   * What happens when a manager is promoted to admin mid-session? (Current JWT retains old role until refresh; new role applied on next login/token refresh)
