## Story ID
   * ID Format: US_005

## Story Title
   * Session Management and Token Refresh

## Description
  * As a platform user, I want my session managed securely with automatic JWT token refresh during active use and automatic idle timeout after 30 minutes of inactivity, so that I maintain seamless access while inactive sessions are terminated to prevent unauthorized use.

## Acceptance Criteria
  1. **Given** I am authenticated with a valid JWT, **When** my token approaches expiration during active use, **Then** the system automatically refreshes the token using a refresh token without interrupting my workflow or redirecting to the SSO page.

  2. **Given** I have been idle for 30 minutes (no API calls or UI interaction), **When** the idle timeout expires, **Then** the system invalidates my session, clears the JWT from client storage, and redirects me to the login page with the message "Session expired due to inactivity."

  3. **Given** I click "Logout", **When** the logout request is processed, **Then** the system invalidates my JWT, clears all session data (cookies, localStorage), logs the logout event in the AuditLog, and propagates logout to the SSO provider (single logout).

  4. **Given** session cookies are set, **When** they are created, **Then** cookies are configured with HttpOnly, Secure, and SameSite=Strict flags to prevent XSS and CSRF attacks.

  5. **Given** login and logout events occur, **When** the event is processed, **Then** an AuditLog entry is created with timestamp, user ID, IP address, action type (Login/Logout), and session ID.

## Edge Cases
   * What happens when the refresh token itself expires? (Redirect to SSO login; preserve current URL for post-auth redirect)
   * How does the system handle concurrent sessions from multiple devices? (Allow multiple active sessions; each tracked independently; logout on one device does not affect others)
   * What happens if the client clock is significantly skewed? (Server-side validation of token expiry; do not rely on client-side time)
