## Story ID
   * ID Format: US_002

## Story Title
   * Workday Employee Data Synchronization

## Description
  * As a system administrator, I want employee data (employee ID, name, email, department, manager, role, employment status) automatically synchronized from Workday every 4 hours using incremental delta queries, so that user accounts, organizational structure, and employment status are always current without manual intervention.

## Acceptance Criteria
  1. **Given** the sync schedule is active, **When** the 4-hour cron interval triggers (0 */4 * * *), **Then** the background job authenticates with Workday API using OAuth 2.0 client credentials from Azure Key Vault and fetches delta changes since the last successful sync timestamp.

  2. **Given** the Workday API returns employee delta records, **When** the system processes each record, **Then** it validates required fields (employeeId, name, email) and data types (email format, status enum), quarantining invalid records with error details while continuing to process valid records.

  3. **Given** a new employee is found in the Workday response, **When** the employee ID does not exist in the User table, **Then** the system creates a new user account, assigns the correct role from Workday data, and queues a welcome email with platform introduction.

  4. **Given** an existing employee record has changed, **When** the system detects differences in name, email, department, manager, or role, **Then** it updates the User table within a database transaction ensuring atomicity, and invalidates relevant Redis caches (team_roster for manager changes).

  5. **Given** an employee is terminated in Workday, **When** EmploymentStatus is 'Inactive', **Then** the system soft-deactivates the user account (retain audit history), revokes all active sessions, and logs the deactivation action.

  6. **Given** a sync completes, **When** all records are processed, **Then** the system logs a SyncLog entry (records processed, created, updated, deactivated, quarantined, duration) and sends an email summary to admin users.

  7. **Given** the Workday API authentication fails, **When** all 3 retry attempts are exhausted, **Then** the system logs a failure record, sends an urgent alert to admins, and schedules retry in 1 hour.

## Edge Cases
   * What happens when the Workday API returns a 429 rate limit response? (System respects Retry-After header, implements exponential backoff, continues sync after cooldown)
   * How does the system handle a partial sync failure mid-transaction? (Database transaction rolls back; failed batch quarantined; admin alerted; successful records from prior batches are committed)
   * What happens when manager ID in Workday references an employee not yet synced? (Defer manager relationship update; log dependency; resolve on next sync cycle)
   * How does the system handle Workday returning duplicate employee IDs? (Unique constraint prevents duplicate inserts; quarantine duplicate record with error)
